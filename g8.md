Here’s a **deliberately nasty grammar** designed to exercise **nullable symbols, chains of nullables, ε-only productions, and mixed terminal/nonterminal prefixes**.

## Challenging Test Grammar (ε-heavy)

### Grammar

```
S  → A B C D
A  → a A | ε
B  → b B | ε
C  → c | ε
D  → d D | e
```

---

## Why this grammar is tricky

### Nullable nonterminals

* `A` is nullable (ε or many `a`s)
* `B` is nullable (ε or many `b`s)
* `C` is nullable (ε or `c`)
* `D` is **not** nullable, but starts with a terminal

This creates:

* **Long nullable prefixes**
* Multiple paths where FIRST(S) must accumulate terminals from *later* symbols

---

## Expected FIRST sets (for validation)

```
FIRST(A) = { a, ε }
FIRST(B) = { b, ε }
FIRST(C) = { c, ε }
FIRST(D) = { d, e }

FIRST(S) = { a, b, c, d, e }
```

Why?

* A may vanish → look at B
* B may vanish → look at C
* C may vanish → look at D
* D is not nullable → stop there

## Extra evil version (mutual nullability)

If you really want to push it further:

```
S → X Y Z
X → A | ε
Y → B | ε
Z → C

A → a A | ε
B → b B | ε
C → c | D
D → ε
```

### Expected behavior

* `X`, `Y`, `A`, `B`, `D` are nullable
* `C` is nullable *and* can produce a terminal
* FIRST(S) must walk **three nullable layers deep**

Expected:

```
FIRST(S) = { a, b, c, ε }
```

---

## Nuclear option (use this to find bugs)

```
S → A B
A → ε
B → C
C → ε
```

* Every symbol is nullable
* FIRST(S) **must contain ε**
* Many broken implementations forget to propagate ε all the way up

Expected:

```
FIRST(S) = { ε }
```
